## Work Done

### L2C 0i8d 정책 추가

- `ChampSim_FDIP`의 L2C partition에 `0i8d` 모드를 추가했다.
- `0i8d`에서는 instruction-origin access가 L2C tag lookup/fill/miss stat을 남기지 않고 LLC로 전달되도록 했다.
- 작은 workload로 기능을 검증했다.
  - `shared`에서는 L2C `LOAD_I` access/miss가 관측된다.
  - `0i8d`에서는 L2C `LOAD_I` access/miss가 0으로 내려간다.
  - LLC `LOAD_I`는 유지되어 instruction line이 LLC에서 처리되는 것을 확인했다.

### Run script 정리

- `scripts/run.sh`의 help를 `Actions`, `Location parameters`, `Run parameters`로 나눠 읽기 쉽게 정리했다.
- `-C <dir>` 옵션을 추가해 ChampSim directory를 선택할 수 있게 했다.
- `-D` 옵션을 추가해 상세 default path는 필요할 때만 확인하도록 했다.
- `-v` 옵션은 실제 역할이 없어 제거했다.
- `-f <mask>`를 FTQ bitmap 방식으로 변경했다.
  - `0x01`: ftq0/off
  - `0x02`: ftq2
  - `0x04`: ftq4
  - `0x08`: ftq16
  - `0x10`: ftq32
  - `0x20`: ftq64
  - `0xff`: all
- `-L2C <mask>` 정책을 재정의했다.
  - `0x01`: shared
  - `0x02`: 0i8d
  - `0x04`: 2i6d
  - `0x08`: 4i4d
  - `0x10`: 6i2d
  - `0x1f`: all
- `-s` summary bitmask에 두 비트를 추가했다.
  - `0x40`: metrics.csv를 (재)생성만 한다. 다른 표/그래프 비트는 metrics.csv를 읽기만 하고 자동으로 만들어주지 않으며, 없으면 바로 에러를 낸다(예전의 조용한 auto-generate 제거).
  - `0x80`: `delta_grid.py`를 호출해 L2C partition delta 그래프+CSV를 생성한다. `shared` 외 policy가 있어야 의미가 있다.
- output 경로에 `<l2c_policy>` 단계가 추가되어 `raw/fdip_<ftq>/<policy>/...`, `summary/fdip_<ftq>/<policy>/metrics.csv` 구조가 되었다.

### Parser와 그래프 보강

- `parser/parse_outputs.py`
  - ChampSim 로그의 `====ROB Stall Breakdown====` 블록(`ADDR_TRANS`/`REPLAY_LOAD`/`NON_REPLAY_LOAD`)을 파싱해 `rob_stall_*` 컬럼으로 남겼다.
  - 이를 기존 frontend/backend stall breakdown과 조합해 분석에 바로 쓸 수 있는 파생 지표 세 개를 추가했다.
    - `frontend_instruction_fetch_stall`: L1I miss로 인한 frontend stall.
    - `backend_instruction_stall`: LQ/SQ stall + `ADDR_TRANS`/`REPLAY_LOAD` (주소변환·재시도 등 instruction-side와 얽힌 backend stall).
    - `backend_data_stall`: `NON_REPLAY_LOAD` (순수 data load 대기로 인한 backend stall).
  - 이 세 지표가 `delta_grid.py`의 stall 관련 축을 구성한다.
- `parser/fdip/cover/make_one_g.py`가 L2C policy별 summary directory 구조를 처리하도록 수정했다. 정책이 여러 개면 2x2 grid 하나로 합쳐서 출력한다.
- `parser/l2c/delta_grid.py`를 추가해 shared 대비 L2C policy별 변화량을 그래프로 볼 수 있게 했다.
  - trace(가로) x metric(세로) 격자 그래프(`l2c_delta_grid.png`)와, FTQ/side(data\|instruction)/policy를 한 줄로 나열한 통합 그래프(`l2c_delta_combined.png`) 두 가지를 만든다.
  - 그래프와 같은 자리에 `l2c_raw_values.csv`(원본 수치), `l2c_delta_pct.csv`(MPKI를 %로 표현한 delta), `l2c_delta_raw.csv`(MPKI를 raw delta로 표현한 delta)를 함께 저장한다.
- `docs/image/l2c_delta_combined.png`를 생성해 trace/FTQ/L2C별 변화 경향을 시각화했다.
- MPKI 변화량은 raw delta와 percent delta 모두 해석이 필요하다는 점을 정리했다(baseline이 0에 가까우면 %가 왜곡됨). 그래프에서는 MPKI는 raw delta, IPC/stall%p는 % 또는 %p로 축을 분리해서 그린다.
- 그래프/CSV 설계가 여러 차례 바뀐 과정과 근거는 `docs/exp/2026_07_14_analysis.md`에 단계별로 남겼다.

## Conclusions

- L2C를 instruction/data로 단순 분리한다고 해서 IPC가 전반적으로 좋아지지는 않았다.
- `sierra.a.6`처럼 instruction-side 압력이 큰 일부 경우에는 instruction way를 더 주는 정책이 유리할 수 있다.
- 반대로 많은 workload에서는 instruction을 보호하면 data-side L2D MPKI가 증가하고, IPC 이득이 작거나 오히려 손해가 난다.
- `0i8d`는 instruction을 L2C에서 완전히 제외하는 control case로 유용하다.
- L2C partition 효과를 판단하려면 `IPC`만 보면 부족하고, 다음 지표를 함께 봐야 한다.
  - L1I/L2I MPKI
  - L1D/L2D MPKI
  - frontend stall
  - backend instruction/data stall
  - L2C 전체 MPKI
- 전체 trace x 전체 FTQ x 전체 L2C 조합을 긴 instruction 설정으로 모두 돌리면 비용이 크다.
  - 전체 343 traces 기준: 약 5.4일
  - 현재 5개 trace group 기준: 약 1.5일
  - 새 8개 L2C test group 기준: 약 4.7일
- 새 8개 group에서 FTQ 일부만 고르는 단기/중기 sweep은 현실적인 비용으로 가능하다.
  - FTQ `0/8/32/64`, 모든 L2C, `w=2M`, `i=30M`, `-p56`: 약 20시간.
