## Work Done

### L2C Partition 장기 실험 진행

- `trace_gtrace_l2c_test.txt`의 8개 trace group, 총 296 traces를 대상으로 장기 실험을 진행했다.
- 실험 설정은 warmup `2,000,000`, simulation `30,000,000`으로 잡았다.
- L2C 정책은 `shared`, `0i8d`, `2i6d`, `4i4d`, `6i2d`를 대상으로 시작했다.
- FTQ는 stage 1(`0`, `4`, `32`)과 stage 2(`2`, `16`, `64`)로 나눠 실행했다.
- stage 1 완료분을 먼저 summary로 변환하고, shared 대비 L2C partition delta를 분석했다.

### Stage 1 결과 분석

- `0i8d`는 instruction을 L2C에서 제거하므로 L2D/L2C pressure를 줄이는 효과가 컸다.
- 하지만 `ftq0`, `ftq4`에서는 instruction miss latency를 충분히 숨기지 못해 IPC가 감소하는 경우가 많았다.
- `ftq32`에서는 instruction-side MPKI 변화가 작아지고, IPC가 data-side MPKI 변화와 더 강하게 맞물리는 경향을 보였다.
- 이 결과를 바탕으로, FTQ가 충분히 크면 instruction을 L2C에서 줄이고 data 공간을 보호하는 정책이 가능성이 있다고 판단했다.
- 반대로 FTQ가 작을 때는 workload에 따라 instruction-side latency가 더 중요해질 수 있으므로, 정적 partition 하나로는 설명이 어렵다고 보았다.

### Workload 특성 분류

- `ftq0`에서는 IPC가 instruction-side 지표를 따라가는 workload와 data-side 지표를 따라가는 workload가 섞여 있었다.
- `delta`, `sierra.a.6`, `tango`는 instruction/frontend-follow 성격이 강한 후보로 분류했다.
- `merced`, `yankee`는 data-follow 성격이 강한 후보로 분류했다.
- `bravo`, `sierra.a.4`, `tahoe`는 한쪽으로 단순하게 설명하기 어려운 mixed/balanced 후보로 정리했다.
- 이 분류는 이후 동적 L2C partition 정책을 만들 때, frontend stall과 data pressure를 함께 보는 방향의 근거가 된다.

### 실패 원인 디버깅

- `fdip_0/4i4d`에서 `yankee_0012`, `yankee_0054`가 `std::bad_alloc`으로 실패했다.
- 단독 재실행과 `-p1` 재현으로, 시스템 메모리 부족이나 병렬 실행 문제가 아니라 특정 trace/policy 조합에서 재현되는 코드 문제임을 확인했다.
- 원본 `ChampSim_FDIP`는 실험 중인 binary를 읽고 있었기 때문에 직접 빌드하지 않고, `ChampSim_FDIP_debug` 복사본에서만 수정과 빌드를 진행했다.
- gdb와 로그를 통해 실패 지점이 cache가 아니라 `PageTableWalker::finish_packet()`임을 확인했다.
- PTW가 upper request를 받을 때 active MSHR/finished/completed/next-step entry 수를 제한하지 않아 backlog가 커지는 것이 핵심 원인으로 판단했다.
- debug copy에서 PTW capacity check와 dependency vector merge 수정을 적용해 실패 trace가 정상 완료되는 것을 검증했다.
- 검증 후 수정사항은 `ChampSim_FDIP`에 commit으로만 반영했고, 진행 중인 실험 보호를 위해 빌드는 하지 않았다.

### L2C Latency 모델 재정의

- L2C partition을 단순히 capacity partition으로만 볼지, lookup latency에도 반영할지 논의했다.
- 실제 하드웨어에서 way 수가 늘어난다고 latency가 정확히 2배가 되는 것은 아니지만, 현재 실험에서는 명시적인 단순 모델이 필요하다고 판단했다.
- 최종 모델은 `way당 1 cycle lookup + fill 1 cycle`로 정리했다.
- shared에서는 instruction/data 모두 전체 8 way를 lookup한다.
- partition 정책에서는 instruction/data가 자기에게 할당된 way만 lookup한다.
- `0i8d`에서는 instruction이 L2C lookup/fill을 건너뛰고 LLC로 내려간다.
- `8i0d`에서는 data가 L2C lookup/fill을 건너뛰고 LLC로 내려간다.
- `hit_latency`라는 이름은 코드 필드로 남아 있지만, 실험 해석상으로는 `lookup_latency`에 가깝다고 정리했다.

### Run/Parser 확장

- `1i7d`, `8i0d` L2C policy를 추가하기 위한 run script와 config 경로를 정리했다.
- `-L2C` bitmask를 확장해 `shared`, `0i8d`, `1i7d`, `2i6d`, `4i4d`, `6i2d`, `8i0d`를 선택할 수 있게 했다.
- `8i0d`는 data-side L2C bypass case로, `0i8d`의 instruction bypass와 대칭되는 control case로 사용할 수 있다.
- L2C delta parser도 bypass policy의 구조적 0 값을 잘 해석하도록 조정했다.

## Observations

- L2C instruction/data partition은 단순히 instruction way를 늘리거나 data way를 늘리는 문제로 끝나지 않는다.
- FTQ가 작을 때는 instruction miss latency가 IPC를 강하게 흔들 수 있고, FTQ가 커지면 data-side pressure가 더 잘 드러난다.
- `0i8d`는 data 공간을 보호하는 가장 강한 정책이지만, instruction latency를 감당할 수 있는 frontend 조건이 함께 필요하다.
- PTW backlog 문제는 L2C partition 자체의 직접 버그라기보다, 특정 trace/policy가 원래 있던 PTW capacity 문제를 드러낸 사례에 가깝다.
- L2C partition 실험을 공정하게 비교하려면 capacity뿐 아니라 lookup latency와 bypass latency를 명시적으로 모델링해야 한다.
