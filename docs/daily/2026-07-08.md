## ChampSim_FDIP 변경

`ChampSim_FDIP`는 어제 정리한 FDIP base 위에서 추가 계측을 넣었다. 기준은 `ChampSim_FDIP` commit `526782c331353df892e6620a82bd770ce1f82f9c`이며, 이후 변경사항은 `ChampSim_FDIP_new`와 비교하면 확인할 수 있다.

핵심 구현:

- `channel::request`에 `is_instr_fetch` 플래그를 추가했다.
- L1I에서 하위 캐시로 내려보내는 miss는 instruction-fetch origin으로 표시한다.
- L2C와 LLC에서는 이 플래그를 이용해 hit, miss, MSHR merge, MSHR return을 instruction/data origin별로 나누어 기록한다.
- cache block에도 `is_instr_fetch`를 저장해서, prefetch된 라인이 나중에 useful/useless가 될 때 그 라인의 origin을 추적할 수 있게 했다.
- `plain_printer.cc` 출력에 `LOAD_I`, `LOAD_D`, `RFO_I`, `RFO_D`처럼 origin별 통계를 추가했다.
- split된 average miss latency와 prefetch useful/useless 통계도 함께 출력하도록 했다.

이 변경은 L2C/LLC에서 기존 `LOAD`, `RFO`, `PREFETCH`가 instruction과 data를 섞어서 보여주던 문제를 줄이기 위한 것이다. 앞으로는 `l2i_mpki`, `l2d_mpki`, `lli_mpki`, `lld_mpki`를 따로 계산할 수 있다.

## Parser / Run Script 변경

root parser와 run script도 analysis flow에 맞게 수정했다.

- `parser/parse_outputs.py`
  - cache log parser가 `LOAD_I`, `LOAD_D`, `RFO_I`, `RFO_D`처럼 `_I`/`_D` suffix를 읽을 수 있게 변경했다.
  - `l1i_mpki`를 metrics에 추가했다.
  - L2C/LLC origin split 기반으로 `l2i_mpki`, `l2d_mpki`, `lli_mpki`, `lld_mpki`를 계산하도록 했다.
  - 옛날 binary로 만든 로그처럼 `_I`/`_D` 줄이 없으면 해당 값은 빈 값으로 남기도록 했다.

- `parser/summary.py`
  - full summary table에 L1I MPKI를 추가했다.
  - `--minimal` 모드에서는 I/D split MPKI 중심 표를 출력하게 했다.
  - `--fdip` 모드에서는 FDIP coverage와 L1I Miss 비율만 보는 표를 출력하게 했다.

- `scripts/run.sh`
  - 실험 대상 ChampSim directory를 `ChampSim_FDIP_new`로 바꿔서, 기존 장기 실행 중인 process에 영향을 주지 않고 새 계측 binary를 따로 빌드/실험할 수 있게 했다.
  - `-s` summary mask가 10진수와 `0x` 형식의 16진수를 모두 받도록 했다.
  - summary bit를 확장했다.
    - `1`: 기본 summary table
    - `2`: FDIP cover 분석
    - `4`: hit map 분석
    - `8`: I/D split MPKI 중심 minimal table
    - `0x10`: FDIP coverage table (FDIP Cov, L1I Miss + OnChip/OffChip MPKI)

- `parser/summary.py` 후속 변경
  - `--fdip` 표에 `OnChip MPKI`/`OffChip MPKI` 컬럼을 추가했다. 두 값(`on_chip_traffic_mpki`, `off_chip_traffic_mpki`)은 원래 `metrics.csv`에는 있었지만 어떤 표에도 노출되지 않았는데, FDIP 자체 효율(coverage/miss)과 온칩·오프칩 트래픽 부담을 한 표에서 같이 보기 위해 FDIP table로 옮겼다. `l2c_mpki`/`llc_mpki`와 달리 LOAD/RFO뿐 아니라 PREFETCH/WRITE/TRANSLATION까지 포함한 LLC TOTAL 기준이라는 점을 `docs/champsim_summary_analysis.md`에 같이 적었다.

## FDIP_ideal 추가 지표 조사

`FDIP Cov`/`L1I Miss` 외에 `ChampSim_FDIP_ideal`(원본 fork)에 우리가 놓친 FDIP 전용 지표가 더 있는지 확인했다.

- `ChampSim_FDIP_ideal`과 `ChampSim_FDIP`의 `cache_stats.h`/`plain_printer.cc`를 직접 diff한 결과, FDIP 관련 필드는 완전히 동일했다. 즉 `fdip_l1i_hit_covered`, `fdip_l1i_hit_non_prefetch`, `fdip_l1i_late_prefetch_merge`, `fdip_l1i_merge_non_prefetch`, `fdip_l1i_miss` 5개 카운터(=FDIP Cov, L1I Miss의 근거)가 전부이고, 놓친 게 없음을 확인했다.
- 코드에 남아있지만 죽어있는 것들도 확인했다: `ooo_cpu.h`의 `ftq_accumulated_issued_count`("Adaptive FTQ"용)는 어디서도 출력되지 않는 미사용 필드이고, `ChampSim_FDIP_ideal/script/`의 legacy 분석 스크립트(`mispre_tlb.py`, `fetch_latency.py`, `stlb_miss_rank.py`)가 찾는 `[FDIP Post-Mispred]`, `[Fetch Latency]` 로그 포맷은 현재 두 저장소 소스 어디에도 없다 — 파싱 불가능한 죽은 포맷이다.
- 결론: 현재 노출 중인 FDIP Cov/L1I Miss가 FDIP 고유 지표의 전부이며, 별도 계측을 새로 추가하지 않는 한 더 뽑아낼 게 없다.

## Research Direction

앞으로의 핵심 질문은 L2C와 LLC를 instruction stream과 data stream이 함께 사용할 때, 둘 사이의 간섭이 성능 문제를 만드는지 확인하는 것이다.

오늘은 그 질문을 보기 위한 첫 단계로 L2C/LLC의 instruction/data MPKI를 분리해서 확인했다. 다음 단계에서는 다음을 판단해야 한다.

- FDIP의 FTQ size가 커질수록 L2I/LLI MPKI는 줄어드는가.
- 그 과정에서 L2D/LLD MPKI 또는 data-side latency가 증가하는가.
- instruction prefetch가 data working set을 밀어내는 흔적이 있는가.
- trace group별로 instruction pressure가 큰 workload와 data pressure가 큰 workload가 어떻게 나뉘는가.
- MPKI 외에 prefetch useful/useless split, MSHR merge split, average miss latency split 중 어떤 지표가 간섭을 가장 잘 설명하는가.

오늘까지는 환경설정, parser, summary 출력 기반이 거의 정리되었다. 이제는 큰 실험 결과가 완료되는 대로 I/D split MPKI와 FDIP coverage를 함께 보고, 간섭을 설명할 수 있는 추가 지표를 좁혀가면 된다.
