### 0716 결과 분석

- `260716_1305_w10_i20_latency_test`를 `260714_2030_w20_i300_l2c_partition`와 비교했다.
- 새 모델에서는 대부분의 partition 정책이 shared 대비 IPC 이득을 보였다.
- MPKI 변화 방향은 이전 run과 크게 다르지 않았기 때문에, IPC 반전의 주된 원인은 miss-rate 변화가 아니라 latency 모델 변화라고 판단했다.
- 특히 static partition에서 자기 partition way만 lookup하도록 한 효과가 크고, shared가 instruction/data 모두 8-way lookup을 지불하는 모델이 강한 영향을 준 것으로 보았다.
- `6i2d`가 모든 trace에서 가장 좋은 정책으로 나타났는데, 이는 data access 빈도가 높고 data search latency를 크게 줄이는 효과가 capacity 손해를 압도했기 때문이라고 해석했다.

### `8i0d` 실패 원인 분석

- `8i0d`는 data way가 0인 정책이므로 data access가 L2C를 bypass해야 한다.
- smoke test에서 `8i0d`만 모든 trace에서 `finish_packet cannot find a matching entry` assertion으로 실패했다.
- 원본 `ChampSim_FDIP`는 실험 보호를 위해 바로 수정/빌드하지 않고, `ChampSim_FDIP_debug` 복사본에서만 재현과 수정 검증을 진행했다.
- 문제는 L2C를 bypass한 data request가 lower-level response를 받을 때 L2C MSHR에 matching entry가 없는 경우로 좁혀졌다.
- 추가로 bypassed writeback이 `response_requested=true` 기본값을 유지하면서 lower WQ로 내려가는 문제도 확인했다.

### `8i0d` Fix 반영

- bypassed demand/prefetch는 L2C에 fill되지는 않지만, lower-level completion을 안전하게 drain하기 위해 MSHR을 유지하도록 수정했다.
- bypassed writeback은 response가 필요 없으므로 `response_requested=false`로 내려가도록 수정했다.
- debug copy에서 `bravo.a_0000` 및 `bravo.a_0000~0002`를 실행해 abort가 사라지는 것을 확인했다.
- L2C data-side access는 0으로 유지되고, LLC data-side traffic은 정상적으로 관측되어 `8i0d` data bypass 경로가 의도대로 동작함을 확인했다.
- 검증 후 수정사항을 `ChampSim_FDIP` commit `15d240d`로 반영했다.
- 이후 debug copy와 `codex_debug*` 출력물은 정리했다.

## Observations

- way 기반 lookup latency 모델은 L2C partition 결과를 크게 뒤집을 만큼 강한 가정이다.
- static partition은 capacity 적응성을 잃지만, 현재 모델에서는 lookup 대상 way 수를 줄이는 이득이 매우 크게 반영된다.
- `6i2d`가 모든 trace에서 1등이 된 것은 흥미롭지만, 이것을 곧바로 일반적인 cache 설계 결론으로 해석하기에는 조심해야 한다.
- shared L2C의 장점은 workload phase에 따라 instruction/data가 cache capacity를 유연하게 나눠 쓰는 데 있는데, 현재 search latency 모델에서는 그 장점보다 lookup latency penalty가 더 크게 작동한다.
- 따라서 다음 단계에서는 static partition 자체보다, replacement policy가 instruction/data 비율을 동적으로 조정하는 방향도 연구 후보가 될 수 있다.
