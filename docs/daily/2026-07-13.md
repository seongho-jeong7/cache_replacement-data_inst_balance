## Work Done

### Frontend/backend stall 분석 정리

`frontend_stall_test` 결과를 보면서 FDIP의 IPC 개선이 workload마다 왜 다르게 나타나는지 분석했다.

- `FF-L1I`, `NoFetch`, `FF-Decode`를 구분해서 보기 시작했다.
- 기존 `ROB_STALL`과 새로 본 `FF-Decode`가 같은 의미가 아니라는 점을 정리했다.
  - `FF-Decode`는 fetch/decode 쪽 buffer가 막혀 frontend 승격이 실패한 비율이다.
  - `ROB_STALL`은 dispatch 이후 backend 자원이 막힌 비율이다.
- 12개 trace group을 frontend-limited, backend-limited, data/memory-limited, mixed 성격으로 나누어 보기 시작했다.
- FDIP로 L1I/L2I miss가 크게 줄어도 IPC가 오르지 않는 경우는 backend나 decode pressure가 이미 강한 경우가 많다는 가설을 세웠다.

### L2C I/D static partition 구현

`ChampSim_FDIP`에 L2C way를 instruction/data로 정적으로 나눠 쓰는 기능을 추가했다.

- `champsim_config.json`의 `L2C`에 partition 설정을 추가했다.
  - `partition: "shared"`
  - `partition: "static"`
  - `instruction_ways`
  - `data_ways`
- `config/instantiation_file.py`, `inc/cache_builder.h`, `inc/cache.h`, `src/cache.cc`를 수정해 L2C partition 설정이 simulator에 전달되도록 했다.
- `src/cache.cc`에서 L2C fill/victim 선택 시 instruction-origin line과 data-origin line이 지정된 way 범위에 들어가도록 처리했다.
- 현재 구현은 L2C를 물리적으로 두 개로 나누는 방식이 아니라, 하나의 L2C 안에서 way 범위만 나누는 방식이다.

이 기능은 L2C 내부에서 instruction line과 data line이 서로 capacity를 밀어내는지 보기 위한 실험 장치다.

### Run script 개선

L2C partition 실험을 편하게 반복할 수 있도록 `scripts/run.sh`를 크게 개선했다.

- `-L2C` 옵션을 추가했다.
  - `0x1`: shared
  - `0x2`: 2:6
  - `0x4`: 4:4
  - `0x8`: 6:2
  - `0xf`: 전체
- `-b`를 실행하면 선택 mask와 관계없이 L2C policy별 binary 4개를 모두 만든다.
  - `champsim_l2cshared`
  - `champsim_l2c2i6d`
  - `champsim_l2c4i4d`
  - `champsim_l2c6i2d`
- 실행 job은 `(L2C policy, FTQ size, trace)` 단위로 만들어 `-p` 병렬 슬롯에 함께 태운다.
- 결과 경로를 policy별로 나누었다.
  - `raw/fdip_0/shared/...`
  - `raw/fdip_0/2i6d/...`
  - `raw/fdip_0/4i4d/...`
  - `raw/fdip_0/6i2d/...`
- summary도 policy별로 저장하도록 했다.

이전에는 partition별로 config를 바꾸고 빌드와 실행을 반복해야 했지만, 이제는 한 run 안에서 여러 L2C policy를 섞어 실행할 수 있다.

## Observations

오늘 가장 중요한 관찰은 `L1I MPKI` 자체보다 `L2I MPKI`와 frontend stall이 더 직접적인 설명력을 가질 수 있다는 점이다.

L2C partition을 바꿔도 L1I MPKI는 크게 변하지 않았다. 하지만 L1I miss가 L2C에서 hit하는지, L2C에서도 miss가 되어 더 긴 latency를 만드는지는 크게 달라졌다. 이 차이가 `L2I MPKI`와 `FF-L1I%`에 나타났고, sierra.a.6에서는 IPC 차이로도 이어졌다.

따라서 앞으로의 연구 질문은 단순히 "instruction miss가 줄었는가"가 아니라 다음처럼 바뀌어야 한다.

- instruction miss가 어느 cache level에서 잡히는가.
- instruction line이 data line을 밀어내는가, 또는 data line이 instruction line을 밀어내는가.
- 그 변화가 L2C 내부에서 끝나는가, LLC/off-chip까지 번지는가.
- MPKI 변화가 실제 stall/latency 감소로 이어지는가.
